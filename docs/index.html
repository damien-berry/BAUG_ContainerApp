08 / Session Groups

Session Groups: The Missing Piece

Explain how Azure Container Apps Sandboxes are organised and why the Build demo appeared to magically inherit access to Teams, Calendar and Email.

Azure Container Apps sandboxes overview

User
↓
Session Group
├─ Connectors
├─ Identity
├─ Storage
├─ Policies
└─ Environment
↓
Session 1
Session 2
Session 3
  • Sessions belong to a Session Group.
  • Session Groups define shared configuration.
  • Connectors are attached at the Session Group level.
  • New sessions inherit configuration automatically.
  • This explains how agents in the Build demo immediately accessed Teams and Calendar.
When I first watched the demo I assumed every sandbox was individually configured. After reading the documentation it appears Session Groups own the shared configuration and sessions inherit it. This is likely how Microsoft enables rapid sandbox creation while preserving a security boundary.

09 / Snapshots and State Management

Snapshots and State Management

Explain why ACA Sandboxes are more than disposable containers.

Azure Container Apps sandboxes snapshots and state management

Create Session
↓
Install Tools
↓
Authenticate
↓
Load Files
↓
Create Snapshot
↓
Resume Later
↓
Clone Session
  • Sessions can be snapshotted.
  • State can persist across executions.
  • Sandboxes can be resumed instead of recreated.
  • Existing sessions can be cloned.
  • Long-running agent workflows become possible.
Traditional containers are usually disposable. ACA Sandboxes introduce the concept of persistent state and snapshots. This starts to look less like container orchestration and more like a managed runtime designed specifically for agents.

10 / Trust Through Egress Control

Trust Through Egress Control

Explain why ACA Sandboxes may be suitable for executing AI-generated code.

Azure Container Apps sandboxes egress policies

AI Agent
↓
Sandbox
↓
Egress Policy
├─ Allow Graph
├─ Allow GitHub
├─ Allow Jira
├─ Allow Internal API
└─ Block Everything Else

Internet
  • Sandboxes can restrict outbound network access.
  • Access can be explicitly allow-listed.
  • AI-generated code can be prevented from calling arbitrary endpoints.
  • Security is enforced by the platform rather than relying entirely on prompts.
  • Policies apply regardless of what code the agent generates.
This was the point where ACA Sandboxes became genuinely interesting. The challenge with agent-generated code is not writing the code. The challenge is controlling what that code can access. Egress policies provide a platform-level security boundary. Instead of trusting the model, we constrain the runtime.

11 / Identity, Connectors and Trust

Identity, Connectors and Trust

What we know from the Build demo and documentation

What We Observed

User Identity
↓
Session Group
↓
Connectors
↓
Enterprise Systems

Examples from the Build demo: Calendar, Teams, Email.

  • Connectors were demonstrated accessing user-centric services.
  • Sessions appear to inherit connector configuration from Session Groups.
  • Connector configuration is not repeatedly configured per session.
  • User context appears to be an important part of the runtime model.

Questions Raised

  • Who grants consent?
  • Who owns the token?
  • What is the isolation boundary?
  • How is access audited?
  • What happens when tokens expire?
  • Can User A access User B data?
  • Can application identities be used instead of user identities?

Observation

The ACA Sandbox model appears to be identity-aware, not merely compute-aware.

The Build demo and documentation show identity, connectors and session groups working together to provide access to enterprise systems.

At this stage it is clear identity is important to the platform, but the precise security and tenancy model requires deeper investigation.

When I first watched the Build session I assumed the sandbox itself was being configured with access to Teams and Calendar. After reviewing the documentation, it appears Session Groups own shared configuration and sessions inherit that configuration. This is a more enterprise-friendly model, but it raises important architecture questions around identity, consent, auditing and tenancy boundaries.

12 / My Take

My Take

Damien's Interpretation - What I think Microsoft is really building

ACA = Compute

Azure Container Apps is the execution platform.

The runtime provides:

  • Scale
  • Isolation
  • Startup performance
  • Session lifecycle

Observation: This is the foundation layer.

Sandboxes = Isolation

The most significant announcement in the session.

Sandboxes provide:

  • Secure code execution
  • Session persistence
  • Snapshots
  • Session groups
  • Egress controls

Observation: This feels like a new runtime category designed specifically for agent workloads.

Connectors = Capability

Agents become useful when they can interact with systems.

Examples:

  • Email
  • Calendar
  • Teams
  • GitHub
  • Internal APIs

Observation: The sandbox runs the agent. The connectors allow it to perform work.

Identity = Trust

Identity appears throughout the architecture.

Examples:

  • Connector access
  • User context
  • Session ownership
  • Consent
  • Auditing

Observation: The runtime appears to be identity-aware rather than simply container-aware.

The Big Shift

Traditional Thinking

Model
↓
Application
↓
Database

Agent Thinking

Model
↓
Runtime
↓
Identity
↓
Tools
↓
Systems

My biggest takeaway from Build 2026 was that Microsoft is focusing less on the model and more on the runtime.

The difficult part of enterprise AI is not generating tokens. The difficult part is safely running agents that can interact with real systems.

I expected a presentation about agents. Instead I think I watched a presentation about runtimes. The demo was interesting. The announcement was Sandboxes. The documentation reinforced that identity, session management, persistence and network controls are becoming first-class concerns. The models will continue to improve. The runtime is where most of the engineering effort will live.

13 / Where I Would Use ACA Sandboxes

Where I Would Use ACA Sandboxes

Not every workload needs a sandbox

Strong Fit

  • AI generated code
  • User supplied scripts
  • Browser automation
  • Multi-tenant AI platforms
  • Interactive coding environments

Observation: The workload is not fully trusted.

Possible Fit

  • Enterprise assistants
  • Graph-connected agents
  • Research agents
  • Long-running workflows

Observation: More operational guidance is needed.

Weak Fit

  • Traditional APIs
  • Business applications
  • Integration services
  • Known code paths

Observation: The sandbox solves a problem that may not exist.

Just because something can run in a sandbox does not mean it should. The strongest use cases involve workloads where the code, user or execution path is not fully trusted.

ACA Sandboxes become most valuable when code execution cannot be fully trusted.

I don't think Sandboxes are competing with Functions. I think they are solving a different problem. The less confidence I have in what will execute, the more valuable the sandbox model becomes. This was the slide that helped me understand where the product fits.

15 / What Would Make Me Use This Tomorrow?

What Would Make Me Use This Tomorrow?

Ready Today

  • Isolation
  • Session management
  • Snapshots
  • Egress control
  • Multi-tenancy

Need More Information

  • Identity model
  • Connector governance
  • Cost at scale
  • Monitoring
  • Auditability

Not Yet

  • Core business workflows
  • Mission critical integrations
  • High assurance automation
I wouldn't adopt Sandboxes because they are new. I would adopt Sandboxes when I need to execute workloads that I do not fully trust.
The product solves a real problem. The question is not whether it works. The question is whether your workload actually needs the isolation model it provides.
