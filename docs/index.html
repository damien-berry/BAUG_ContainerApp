The problem

The real challenge for AI agents is not only the model. It is the operating environment that makes the agent responsive, secure, observable, and affordable in production.

  • Runaway loops and repeated tool calls can quickly inflate cost.
  • Untrusted code paths need strong isolation.
  • Long-running tasks need persistence and fast resume behavior.

What Microsoft showed

Microsoft’s BRK221 narrative focused on runtime-first delivery: sandboxed execution, fast cold starts, connector inheritance, and long-running operational continuity.

  • Multi-agent orchestration and voice/text interactions.
  • Fast sandbox creation and resume behaviour.
  • Secure connectors for real enterprise data paths.

Why runtime matters

The runtime is the layer that turns an AI model into a durable system. It determines whether the agent is usable, trustworthy, and cost-effective when real users depend on it.

Why Azure Container Apps

Azure Container Apps is a practical platform for this talk because it combines serverless execution, easy integration, and the kind of operational primitives that agent systems need.

  • Simple deployment path for container-based agent workloads.
  • Good fit for experimentation, iteration, and later production hardening.
  • Room to add observability, connectors, and scaling decisions over time.

What I would use today

Today I would use this pattern for controlled prototypes, internal assistants, and bounded automation where runtime behaviour matters more than raw model novelty.

What I am sceptical about

The hype is real, but the operational story still needs discipline. I want to keep the talk grounded in what stands up in production rather than what looks impressive on stage.

  • Agent loops are still expensive if not tightly governed.
  • Tool access and sandbox boundaries must be explicit.
  • Long-lived agent state needs careful design review.
